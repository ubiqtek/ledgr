# Enable Banking API — Personal Open Banking Access

Notes on using [Enable Banking](https://enablebanking.com) to pull live
transaction data directly from Barclays via Open Banking (PSD2), as an
alternative/supplement to manual OFX export download.

Sources: [Quick Start](https://enablebanking.com/docs/api/quick-start/),
[Control Panel docs](https://enablebanking.com/docs/api/control-panel/),
[API reference](https://enablebanking.com/docs/api/reference/),
[FAQ](https://enablebanking.com/docs/faq/). Verified 2026-07-11 — an
earlier version of this doc was generated by Gemini without checking
against these sources and contained inaccuracies (corrected below).

## Access tiers

Enable Banking distinguishes **Sandbox** vs **Production** environments,
and within Production, two activation paths:

- **Unrestricted Mode (Full Activation)** — for public-facing apps.
  Requires a signed contract, completed KYB, privacy policy/ToS URLs,
  and manual review. Not relevant for personal use.
- **Restricted Mode (Account Linking)** — activates an application by
  linking your *own* bank accounts for internal testing. Data retrieval
  is strictly limited to the whitelisted, linked accounts. **No KYB or
  contract required.** This is the relevant tier for pulling only your
  own Barclays transactions into `ledgr`.

(The earlier Gemini-generated version of this doc described this as
"activating live production access without forcing corporate KYB" —
correct in substance, but framed like an undocumented loophole. It's an
official, documented tier, not a workaround.)

## Registration steps

1. Sign in at https://enablebanking.com/sign-in/ (email + one-time link;
   account auto-created on first login) → Control Panel.
2. Control Panel → API applications → register a new application:
   - Choose environment: Sandbox first, then Production (Restricted
     Mode) once working.
   - Private key: let the browser generate one (SubtleCrypto) rather
     than `openssl` manually — it downloads a `<application_id>.pem`
     file after registration. Keep this file out of the repo and any
     synced folder that isn't encrypted at rest.
   - Whitelist the redirect URL(s) the app will use for the OAuth
     callback.
   - In Restricted Mode, add the specific account(s) (sort
     code/account number) to link — this is what scopes production
     access down to just those accounts.

## JWT authentication

Every API call is authorised with an RS256-signed JWT:

Header:
```json
{ "typ": "JWT", "alg": "RS256", "kid": "<application_id>" }
```

Claims:
```json
{
  "iss": "enablebanking.com",
  "aud": "api.enablebanking.com",
  "iat": <now>,
  "exp": <now + up to 86400>
}
```

`exp` can be up to 24h from `iat` (not 1h as the earlier draft claimed).
Sent as `Authorization: Bearer <jwt>` on every request.

## Core flow (Account Information Service)

1. `GET /aspsps?country=GB` — list providers, find Barclays' ASPSP id.
2. `POST /auth` — body: `aspsp`, `access` (scopes + requested validity),
   `redirect_url`, `state`, `psu_type`. Returns a URL to send the user
   (you) to for the bank's own login/consent screen.
3. User completes the bank's login/consent flow; redirected back to
   `redirect_url` with `code` (and `state`) query params, or
   `error`/`error_description` on failure.
4. `POST /sessions` with the `code` → returns `session_id` and the
   linked account list.
5. `GET /accounts/{account_id}/transactions` (also `/balances`,
   `/details`) using the session, with optional date filters.
6. `DELETE /sessions/{session_id}` to revoke access when done.

Base URL: `https://api.enablebanking.com` (the old `api.tilisy.com` is
deprecated).

## Consent lifetime

**Correction from earlier draft:** there is no fixed 90-day cycle.
Session validity (`valid_until`) is set by the caller in the `/auth`
request, capped by each bank's own maximum
(`maximum_consent_validity` field on the `/aspsps` response) — typically
up to **180 days** for most ASPSPs, Barclays included unless stated
otherwise. Enable Banking auto-refreshes the underlying access token
within that window; only when `valid_until` is actually reached does the
PSU need to re-consent (requests then fail with `EXPIRED_SESSION`).

## Security implications

Right now `ledgr`'s model is "nothing leaves the machine" — you download
a file, it gets parsed locally. Enable Banking flips that: you're
routing live read access to a real bank account through a third-party
Finnish fintech's infrastructure, on an ongoing basis (up to 180 days
per consent, auto-renewing the underlying token within that window).
That's a materially bigger trust surface than a one-off CSV/OFX
download, even though the *outcome* (transactions landing in the local
SQLite DB) looks the same.

Specific things to resolve before writing code against this:

- **The private key is a real secret, not just a convenience file.** It
  signs requests that can pull live transaction data for as long as
  consent lasts. Needs `chmod 600` at minimum; must never sit in the
  Google Drive-synced inbox folder alongside statements; macOS Keychain
  is a better fit than a plain file in the config dir if `ledgr` is
  going to hold long-lived credentials at all.
- **Consent validity being ASPSP-controlled (up to 180 days) means a
  leaked key stays dangerous for a long time** — there's no forced
  90-day rotation (that was the earlier draft's mistake). Prefer
  setting `valid_until` deliberately short and re-consenting often over
  maxing it out.
- **Blast radius if Enable Banking itself is compromised or malicious.**
  Trusting their infra not to log/retain transaction payloads (despite
  their no-caching claim), and trusting the OAuth redirect flow to
  Barclays hasn't been spoofed. Standard Open Banking aggregator risk
  (same category as Plaid/TrueLayer/Yapily), not unique to Enable
  Banking, but worth being eyes-open about.
- **Revocation matters more than acquisition.** `DELETE /sessions/{id}`
  should be an easy, rehearsed command — if this ever feels
  compromised, "kill live access" needs to be a known action, not a
  support ticket.
- **Asymmetric failure modes vs OFX import.** A bug in
  `BarclaysOfxParser` can misread a file already in hand; a bug in the
  Enable Banking integration could leak a credential granting an
  attacker ongoing read access to real account activity.

None of this rules the approach out — it's regulated (FIN-FSA) and
Restricted Mode genuinely doesn't require handing over a business
identity — but the stakes justify writing an ADR before implementation
rather than bolting this on quickly.

## Why bother — the manual OFX download UX

Motivation for considering this at all: downloading a Barclays OFX file
manually is painful enough to make live API access genuinely
attractive, not just a novelty. Per account you have to click through
about three layers of Barclays Online Banking to reach the export, the
file lands in `~/Downloads` (not the configured inbox), and it then has
to be moved into `inbox_dir` by hand before `ledgr import` will see it.
Multiply that by however many accounts (current account, credit card
once that parser exists) and it's a real recurring chore — the kind of
friction that live, pull-based sync would remove entirely.

## Open questions before implementing

- Confirm Barclays' actual `maximum_consent_validity` via `/aspsps`.
- Where to store the long-lived private key + session id locally —
  needs to fit ledgr's "nothing leaves the machine, but this one thing
  talks to a hosted aggregator" model. Worth its own ADR if pursued,
  since it's a bigger architectural fork than adding a new
  `StatementParser`.
- Whether Restricted Mode has a cap on number of linked accounts (fine
  for one current account; matters if credit card is added later too).
- Rate limits / costs for Restricted Mode — Enable Banking is
  commercial infrastructure even at this tier; check pricing page
  before relying on it long-term.
